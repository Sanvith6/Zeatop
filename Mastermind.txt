================================================================================
ZETATOP INCIDENT MANAGEMENT SYSTEM (IMS) - MASTERMIND GUIDE
================================================================================
A comprehensive breakdown of the architecture, flow, and implementation of a 
production-grade SRE platform.

1. PROJECT OVERVIEW
--------------------------------------------------------------------------------
Zetatop IMS is a highly resilient, scalable, and observable Incident Management 
System. It is designed to handle high-frequency signals, debounce them into 
actionable incidents, and manage the full lifecycle of an incident with 
strict SRE compliance (RCA, MTTR tracking, etc.).

SYSTEM GUARANTEES
--------------------------------------------------------------------------------
- At-least-once delivery via Redis durable queue patterns.
- Idempotent processing prevents duplicate incidents.
- Eventual consistency between raw signals (MongoDB) and incidents (Postgres).

2. CORE ARCHITECTURE
--------------------------------------------------------------------------------
The system follows a decoupled, service-oriented architecture:

A. CLIENT LAYER:
   - React Dashboard: Real-time monitoring and incident management.
   - Simulation Scripts: Tools to simulate bursts, failures, and load.

B. INGESTION LAYER (FastAPI):
   - Handles incoming signals via POST /api/signals.
   - Responsibilities: Auth (JWT), Rate Limiting, Adaptive Throttling (Backpressure),
     and high-speed ingestion into the Redis queue.

C. QUEUE LAYER (Redis):
   - Acts as a buffer and decoupling agent.
   - Uses BRPOPLPUSH for crash-safe message delivery.
   - Persistent volume and AOF are designed to minimize signal loss using 
     durable queue patterns.

D. WORKER LAYER (Separate Container):
   - Consumes signals from Redis.
   - Performs: Signal classification, Debouncing, and Work Item management.
   - Features: Circuit Breakers (per dependency), Retry logic, and DLQ.

E. STORAGE LAYER:
   - MongoDB: Stores raw signal logs and Dead Letter Queue (DLQ). Optimized for 
     high-write, schema-less signal data.
   - PostgreSQL: Stores structured Work Items (incidents), Audit logs, and RCA.
     Ensures ACID compliance for state transitions.

3. COMPLETE SYSTEM FLOW (End-to-End)
--------------------------------------------------------------------------------
STEP 1: SIGNAL CAPTURE
   - An external system or simulation script hits the API.
   - API validates payload (Pydantic) and checks if the queue is >70% full.
   - If okay, signal is LPUSHed to Redis and 202 Accepted is returned.

STEP 2: RELIABLE DEQUEUE
   - Worker uses BRPOPLPUSH to move signal from 'signals:queue' to 'signals:processing'.
   - If worker crashes, signal remains in 'processing' and is recovered on reboot.

STEP 3: PROCESSING & DEBOUNCING
   - Worker upserts signal to MongoDB (Idempotent).
   - Worker checks Redis Sorted Sets for similar signals in the last 10s.
   - If signals > threshold, it creates/updates a Work Item in PostgreSQL.
   - Partial Unique Indexes in Postgres prevent duplicate active incidents.

STEP 4: OBSERVABILITY & ALERTING
   - Metrics (Prometheus) are updated at every stage (latency, queue depth, error rates).
   - If a new incident is created, an alert webhook is fired.

STEP 5: HUMAN RESOLUTION
   - Operator views incident on React Dashboard.
   - Updates state: OPEN -> INVESTIGATING -> RESOLVED -> CLOSED.
   - RCA (Root Cause Analysis) is MANDATORY for closing.

4. WORKING MECHANISMS (The "Magic")
--------------------------------------------------------------------------------
- CIRCUIT BREAKER: If PostgreSQL/MongoDB fails, the worker opens a circuit. 
  Signals are routed to DLQ (MongoDB) to prevent worker hanging and cascading 
  failures. Circuit breakers are applied per dependency (MongoDB/PostgreSQL) 
  to isolate failures.
- ADAPTIVE THROTTLING: API returns 429 if the queue is backing up (70% capacity) 
  and 503 if full (100%), protecting the system from meltdown.
- IDEMPOTENCY: Every operation is idempotent. Replaying a signal 10 times results 
  in exactly one MongoDB record and one Postgres incident.
- AT-LEAST-ONCE DELIVERY: Redis 'processing' list pattern is designed to 
  minimize signal loss between dequeue and successful storage.

FAILURE SCENARIO (PostgreSQL Down)
--------------------------------------------------------------------------------
- Worker attempts write -> fails -> retries (3x).
- Circuit Breaker opens -> Signals routed to Dead Letter Queue (DLQ).
- Ingestion API remains available (decoupled).
- Recovery resumes processing once Postgres is healthy.

5. IMPLEMENTATION STACK
--------------------------------------------------------------------------------
- BACKEND: FastAPI (Python 3.12+), Pydantic v2.
- WORKER: Asyncio-based worker process.
- FRONTEND: React, Vite, Tailwind CSS, Lucide Icons.
- DATABASE: 
  - PostgreSQL (Relational Work Items)
  - MongoDB (Document-based Signal Logs)
  - Redis (Queue + Cache + Debounce)
- INFRASTRUCTURE: Docker Compose (mult-container setup).
- MONITORING: Prometheus metrics endpoint (/metrics).

TRADEOFF: Redis was chosen for simplicity and sub-millisecond latency. Kafka 
would provide stronger durability and partitioned consumption at extreme scales.

6. SRE COMPLIANCE CHECKLIST
--------------------------------------------------------------------------------
[x] Backpressure (Adaptive Throttling)
[x] Fault Tolerance (Circuit Breakers)
[x] Data Integrity (Idempotent Workers)
[x] Observability (Prometheus + Latency Logs)
[x] Resilience (Crash-safe Redis Queues)
[x] Performance (Async I/O throughout)

7. HOW TO RUN / REPRODUCE
--------------------------------------------------------------------------------
1. Start Containers: `docker-compose up --build`
2. Access Dashboard: http://localhost:3001
3. Access API: http://localhost:8000
4. Run Load Test: `python scripts/load_test.py`
5. Simulate DB Failure: `python scripts/simulate_failure.py`
