================================================================================
ZETATOP SYSTEM ARCHITECTURE
================================================================================

1. HIGH-LEVEL COMPONENT DIAGRAM
--------------------------------------------------------------------------------
[Client] (React Dashboard / Load Scripts)
   |
   | HTTP (JSON Signals)
   V
[Ingestion API] (FastAPI Nodes)
   |
   | LPUSH (Queue)
   V
[Message Queue] (Redis - AOF Persistent)
   |
   | BRPOPLPUSH (Atomic Pop)
   V
[Background Worker] (Separate Process)
   |
   +---[MongoDB] (Raw Signals / DLQ)
   |
   +---[PostgreSQL] (Structured Work Items)

2. SERVICE RESPONSIBILITIES
--------------------------------------------------------------------------------
A. INGESTION API (FastAPI):
   - Authentication (JWT).
   - Rate Limiting (SlowAPI).
   - Backpressure: Returns 429 (Throttled) if Redis queue is >70% full.
   - Durability: Immediate hand-off to Redis.

B. REDIS QUEUE:
   - Buffer: Protects databases from traffic spikes.
   - Persistance: AOF (Append Only File) + Persistent Volume.
   - Debouncing: Uses Sorted Sets to track signal frequency.

C. WORKER SERVICE:
   - Debouncing: Aggregates signals into single incidents.
   - Classification: Maps signal importance to incident severity.
   - Fault Tolerance: Implements per-dependency Circuit Breakers.

D. DATABASES:
   - MongoDB: High-throughput write store for raw event logs.
   - PostgreSQL: Relational store for incident state, RCA, and audit logs.

3. SIGNAL LIFECYCLE (DATA FLOW)
--------------------------------------------------------------------------------
1. INGEST: Signal received -> Validated -> Queued in Redis -> 202 Accepted.
2. DEQUEUE: Worker pulls signal atomically (crash-safe pattern).
3. PROCESS:
   - Logged to MongoDB (Idempotent).
   - Checked against Debounce Window (10s).
   - If threshold met: Create/Update Work Item in Postgres.
4. ALERT: If new incident, trigger webhook/notification.
5. RESOLVE: Operator updates Postgres state via Dashboard.

4. RESILIENCE & RELIABILITY PATTERNS
--------------------------------------------------------------------------------
- AT-LEAST-ONCE DELIVERY: Atomic Redis pop ensures no signal is lost if worker dies.
- CIRCUIT BREAKERS: Prevents worker from hanging on slow/dead DB connections.
- DEAD LETTER QUEUE (DLQ): Signals failing processing are saved to MongoDB for later.
- ADAPTIVE THROTTLING: System slows down producers when internal buffers are full.
- IDEMPOTENCY: Guaranteed by MongoDB upserts and Postgres unique indexes.

5. TECHNOLOGY STACK
--------------------------------------------------------------------------------
- Language: Python 3.12 (Backend/Worker), JS/React (Frontend)
- API: FastAPI + Uvicorn
- Cache/Queue: Redis (6.x+)
- Logs: MongoDB (Document-based)
- Registry: PostgreSQL (Relational)
- Containerization: Docker + Docker Compose
