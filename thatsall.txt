                       ZETATOP - INCIDENT MANAGEMENT SYSTEM
                          COMPLETE PROJECT MANIFEST (SRE)
===============================================================================

1. PROJECT CORE OBJECTIVE
-------------------------
A production-grade, high-throughput incident management platform designed with 
SRE principles at its core. The system handles millions of monitoring signals, 
deduplicates them into incidents, and provides a zero-click observability 
pipeline for real-time health monitoring.

2. TECHNICAL ARCHITECTURE (3-TIER DISTRIBUTED)
-----------------------------------------------
- Ingestion Layer (FastAPI): Receives signals via authenticated REST API.
- Queue Layer (Redis): High-speed, durable boundary for backpressure.
- Processing Layer (Python Workers): Asynchronous consumers that process 
  signals and manage incident state.
- Storage Layer:
    * MongoDB: Stores raw signal data and the Dead Letter Queue (DLQ).
    * PostgreSQL: Stores structured WorkItems (Incidents) and RCA data.
    * Redis: Used for ingestion queue and real-time metric counters.

3. SRE RELIABILITY PATTERNS
---------------------------
- Adaptive Throttling: System automatically rejects requests (429/503) based on 
  Redis queue saturation (Warn at 50%, Critical at 80%).
- Circuit Breakers: Per-dependency isolation (MongoDB/PostgreSQL). If a database 
  slows down, the circuit opens to prevent cascading failures.
- Idempotency: All signals are tracked by a unique 'queue_id'. Re-processing a 
  signal (e.g., after a worker crash) will not create duplicate incidents.
- Dead Letter Queue (DLQ): Signals that fail after 3 retries are moved to a 
  special MongoDB collection for manual audit, preventing data loss.
- Crash Recovery: Workers use Redis BRPOPLPUSH to ensure that signals are never 
  lost mid-processing; stranded signals are recovered on worker restart.

4. OBSERVABILITY STACK (ZERO-CLICK SETUP)
-----------------------------------------
- Prometheus: Automated scraping of two distinct targets:
    * backend:8000 (Ingestion metrics)
    * worker:8001 (Processing & Latency metrics)
- Grafana: Pre-provisioned with "Zetatop SRE Dashboard":
    * Ingestion/Processing Throughput (req/s)
    * Latency Percentiles (p50, p95, p99)
    * Queue Saturation & Depth
    * Dependency Health (Circuit Breaker Status)
- Structured Logging: All services emit JSON-compatible logs with trace IDs 
  for correlation across the distributed pipeline.

5. KEY METRICS TRACKED
----------------------
- ims_signals_ingested_total: Total throughput of the API.
- ims_signals_processed_total: Performance of the worker fleet.
- ims_queue_depth: Leading indicator of system saturation.
- ims_signal_processing_seconds: Histogram tracking end-to-end latency.
- ims_circuit_breaker_state: Binary state of external dependencies.

6. API & ACCESS POINTS
----------------------
- Frontend: http://localhost:3001 (React Dashboard)
- Backend: http://localhost:8000 (FastAPI Swagger: /docs)
- Metrics: http://localhost:8000/metrics (Prometheus Format)
- Prometheus: http://localhost:9090
- Grafana: http://localhost:3002 (admin / Sanvith@123)

7. SECURITY & AUTHENTICATION
----------------------------
- JWT-based Auth: All signal ingestion requires a Bearer Token.
- Role-based: 'sre-intern' demo user pre-configured for testing.
- Secrets: Managed via environment variables in docker-compose.

8. SIMULATION & TESTING
-----------------------
- scripts/simulate_failure.py: Triggers burst traffic and dependency failures.
- scripts/load_test.py: High-concurrency stress test (1000+ signals/sec).
