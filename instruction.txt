## BACKEND REQUIREMENTS

### 1. Signal Ingestion API
- `POST /api/signals` — accepts signal payload:
```json
{
  "component_id": "CACHE_CLUSTER_01",
  "component_type": "cache" | "rdbms" | "api" | "queue" | "nosql" | "mcp",
  "error_message": "Connection timeout",
  "severity": "P0" | "P1" | "P2" | "P3",
  "timestamp": "2026-04-29T10:00:00Z"
}
```
- Rate limit: max 500 requests/sec per IP (use slowapi)
- Push signals into an **asyncio Queue** (in-memory buffer) — never block ingestion
- Return 202 Accepted immediately

### 2. Background Worker
- Consumes from the asyncio queue
- Saves every raw signal to **MongoDB** collection `signals`
- **Debounce logic**: If 100+ signals arrive for the same `component_id` 
  within 10 seconds, create only ONE Work Item in PostgreSQL. 
  Link all signals to that work_item_id in MongoDB.
- If a Work Item already exists and is not CLOSED for that component, 
  just link the new signal to it — do not create a duplicate.

### 3. Alerting Strategy (Strategy Pattern)
- Create an `AlertStrategy` base class
- Implement:
  - `RDBMSAlertStrategy` → P0, pages on-call immediately
  - `CacheAlertStrategy` → P2, sends warning
  - `APIAlertStrategy` → P1
  - `QueueAlertStrategy` → P1
  - `MCPAlertStrategy` → P0
- Log the alert action to console (no real email/SMS needed)
- Factory function maps `component_type` → correct strategy

### 4. Work Item State Machine (State Pattern)
- States: `OPEN → INVESTIGATING → RESOLVED → CLOSED`
- Implement a `WorkItemStateMachine` class with:
  - `transition(new_state)` method
  - Raises `InvalidTransitionError` for illegal transitions
  - **Block CLOSED transition** if RCA is missing or incomplete
- All state transitions must be atomic (use PostgreSQL transactions)

### 5. Work Item APIs
- `GET /api/workitems` — list all, sorted by severity, with Redis cache (TTL 10s)
- `GET /api/workitems/{id}` — detail + linked signals from MongoDB
- `PATCH /api/workitems/{id}/transition` — body: `{ "new_state": "INVESTIGATING" }`
- `POST /api/workitems/{id}/rca` — submit RCA, auto-calculates MTTR

### 6. RCA Object Schema
```json
{
  "incident_start": "2026-04-29T09:00:00Z",
  "incident_end": "2026-04-29T10:30:00Z",
  "root_cause_category": "Infrastructure" | "Code Deployment" | 
                          "Configuration Change" | "External Dependency" | 
                          "Unknown",
  "fix_applied": "Restarted cache cluster and flushed stale keys",
  "prevention_steps": "Add automated restart policy and alerting threshold"
}
```
- All fields are mandatory
- MTTR = incident_end - incident_start (stored in minutes)
- After RCA submitted, automatically transition Work Item to RESOLVED

### 7. Observability
- `GET /health` — returns `{ "status": "ok", "uptime": seconds }`
- Every 5 seconds, print to console:



[METRICS] Signals ingested: 320/sec | Queue depth: 45 |
Active incidents: 7 | Avg MTTR: 32 min

### 8. Retry Logic
- All PostgreSQL writes must retry up to 3 times with exponential backoff 
  on transient errors

## DATABASE SCHEMAS

### PostgreSQL — `work_items` table
```sql
id UUID PRIMARY KEY,
component_id VARCHAR,
component_type VARCHAR,
severity VARCHAR,
status VARCHAR DEFAULT 'OPEN',
signal_count INTEGER DEFAULT 1,
created_at TIMESTAMP,
updated_at TIMESTAMP,
rca_id UUID REFERENCES rca(id),
mttr_minutes FLOAT
```

### PostgreSQL — `rca` table
```sql
id UUID PRIMARY KEY,
work_item_id UUID REFERENCES work_items(id),
incident_start TIMESTAMP,
incident_end TIMESTAMP,
root_cause_category VARCHAR,
fix_applied TEXT,
prevention_steps TEXT,
submitted_at TIMESTAMP
```

### MongoDB — `signals` collection
```json
{
  "_id": ObjectId,
  "component_id": "string",
  "component_type": "string",
  "error_message": "string",
  "severity": "string",
  "timestamp": "datetime",
  "work_item_id": "string or null"
}
```

### Redis Keys
- `dashboard:workitems` → JSON list of active work items (TTL: 10s)
- `debounce:{component_id}` → work_item_id (TTL: 10s, for debounce window)

## FRONTEND REQUIREMENTS (React + Vite)

### Pages / Components

**1. Live Dashboard (`/`)**
- Table of active Work Items sorted by severity (P0 first)
- Columns: ID, Component, Type, Severity, Status, Signal Count, Created At
- Color coded rows: P0=red, P1=orange, P2=yellow, P3=green
- Auto-refreshes every 5 seconds
- Status badge with color

**2. Incident Detail (`/incident/:id`)**
- Shows Work Item info at top
- Tab 1: Raw Signals — table from MongoDB (signal list)
- Tab 2: Timeline — status history
- Transition buttons: "Start Investigating", "Mark Resolved"
- Link to RCA form if status is RESOLVED

**3. RCA Form (`/incident/:id/rca`)**
- Date-time pickers for Incident Start and End
- Dropdown for Root Cause Category
- Textarea for Fix Applied
- Textarea for Prevention Steps
- Submit button → calls `POST /api/workitems/{id}/rca`
- Shows calculated MTTR after submission
- Validation: all fields required before submit
